# Streamlit Excel Processor (_Анализ / Сальдо / Контракты / Запасы)

Это Streamlit‑приложение собирает входные Excel‑файлы в единый файл вида `_Анализ …` и затем выполняет одну или несколько обработок:

- `Сальдо` → создаёт листы `сальд` и `сальд (2)` (по префиксам компаний).
- `Контракты` → создаёт лист `контр` (по префиксам компаний) и удаляет исходные `Wd/Md`.
- `Запасы` → дописывает формулы/итоги на листах ОСВ 1310/1320/1330.

Код: `app.py`.

## Быстрый старт (локально)

Требования:

- Python 3.10+ (рекомендуется 3.11).
- Для поддержки `.xls` локально: Windows + установленный Microsoft Excel.

Запуск в PowerShell:

## Что такое “Сборка _Анализ”

Приложение работает в 2 этапа:

1. **Сборка**: из набора загруженных файлов создаётся один “рабочий” Excel (`_Анализ …`), куда копируются/добавляются листы.
2. **Обработка**: после сборки становятся доступны чекбоксы `Сальдо / Контракты / Запасы` только если нужные листы реально есть в собранной книге.

В UI сборка запускается кнопкой `Собрать _Анализ`. Во время сборки показывается прогресс и статус по шагам.

---

## Типы входных файлов

Приложение распознаёт файлы по имени:

- `_Анализ*.xlsx/.xlsm` — базовый файл анализа. Если загружен, сборка “вливается” в него.
- `WH_KZ*.xlsx/.xlsm` — источник листов для “W…”.
- `M_KZ*.xlsx/.xlsm` — источник листов для “M…”.
- Все прочие `.xlsx/.xlsm` (и локально `.xls`) считаются ОСВ‑файлами и **автоматически очищаются** перед добавлением.

Ограничение, принятое в логике:

- `_Анализ`, `WH_KZ`, `M_KZ` обычно по одному, но приложение умеет обработать несколько `WH_KZ/M_KZ` через запрос префиксов.
- ОСВ можно загружать много, в том числе с повторяющимися счетами (1210/1710/3310/3510 и др.).

## Префиксы: зачем и как работают

Префикс — это строка перед “суффиксом” листа (например, `М1210`, `МWr`, `МWd`).

Префиксы нужны, потому что в одном `_Анализ` может быть несколько компаний, и листы должны быть различимы.

Правила:

- Для ОСВ‑листов, где определяется 4‑значный счёт, имя листа в `_Анализ` формируется как `<префикс><счет>`.
- Для `WH_KZ`/`M_KZ` импортируемые листы попадают как `<префикс>W`, `<префикс>Wr`, `<префикс>Wt`, `<префикс>Wd` и `<префикс>M`, `<префикс>Mr`, `<префикс>Mt`, `<префикс>Md`.
- Если имя уже занято, добавляется уникализатор: `" (1)"`, `" (2)"`, или вставка счётчика перед суффиксом (см. `make_unique_with_fixed_suffix`).
- Если загружено несколько одинаковых ОСВ‑счетов, приложение **запрашивает префикс для каждого**; можно выбрать “без префикса”, но в рамках одного счета это допустимо только один раз.

Чтобы пользователю было удобно выбирать префиксы, в UI рядом показывается “лейбл компании” (обычно берётся из `A1`).

## Импорт из WH_KZ / M_KZ

При сборке из `M_KZ` (если найдено):

- `Итог` → `M`
- `Контрагенты` → `Mt`
- `Реестр` → `Mr`
- `Договоры` → `Md`

При сборке из `WH_KZ` (если найдено):

- `Итог` → `W`
- `Таблицы` → `Wt`
- `Реестр` → `Wr`
- `Договоры` → `Wd`
- `Кредиты` → `кред` копируется **только если** `AC6 != 0`

Важно про формат “Реестр” (`Wr/Mr`):

- После копирования фиксируется ширина всех колонок (48px) и высота строк (28.8pt).
- Отключается перенос текста (`wrap_text=False`), чтобы Excel не “раздувал” строки.

См. `format_registry_sheet()`.

## ОСВ: автоматическая очистка и определение счета

Любой “непонятно названный” файл ОСВ перед добавлением в `_Анализ` проходит чистку (логика из Colab, адаптированная под Streamlit):

- снимаются группировки/outline,
- размёрдживаются ячейки,
- вычисляется счёт по “2‑й строке” (и некоторым фолбэкам, включая merged‑заголовки),
- вычищаются строки с “мусорными” словами (`OSV_BAD_WORDS`),
- удаляются полные дубликаты строк в блоке данных,
- для 1310/1320/1330 сортировка блока по колонке `G` по убыванию,
- лист переименовывается в номер счета (или добавляется как `OSV`, если счёт определить не получилось).

Ключевые функции:

- `detect_account_from_ws(...)`
- `clean_osv_sheet_inplace(ws)`
- `build_analysis_workbook(...)`

## Обработка “Сальдо” (лист `сальд`)

Функция: `run_code_1(file_bytes)`.

Что делает:

- Ищет листы ОСВ, которые **заканчиваются** на `1210/1710/3310/3510` (суффикс последние 4 символа).
- Группирует листы по префиксу.
- Для каждого префикса создаёт лист `<префикс>сальд` (или `сальд`) и заполняет 3 таблицы:
  - заказчики (1210/3510/сальдо),
  - поставщики (1710/3310/сальдо),
  - общее сальдо.

Все значения в тысячах тенге.

---

## Обработка “Сальдо (2)” (лист `сальд (2)`)

Также создаётся в `run_code_1(...)` как `<префикс>сальд (2)` (или `сальд (2)`).

Особенности:

- Шрифт на листе: Calibri.
- Строки контрагентов: топ‑15 положительных и топ‑15 отрицательных по сальдо.
- После каждого топ‑15 добавляются строки:
  - `ТОП-15` (суммы формулами `SUM(...)`),
  - `прочее` (= `ИТОГО - ТОП-15`),
  - `ИТОГО` (частично числа из pandas, частично формулы без критерия контрагента),
  - пустая строка.
- В столбце `F` заполняется **формула‑комментарий** (“ОК/сомнительный/сомнительно/списание”) по правилам давности оплат/выполнений.
- `G/H` содержат формулы суммы оплат/выполнений за “последние N месяцев” (где `N` в `A2`), но эти формулы не ставятся в блок “общее сальдо”.
- Колонки месяцев:
  - оплаты `J:AG`,
  - выполнения `AI:BF` (для заказчиков).

Важно:

- Листы `Wr/Mr` ищутся по текущему префиксу (включая варианты `Wr (1)`/`Mr (1)`).
- Имена листов внутри формул корректно экранируются.

---

## Обработка “Контракты” (лист `контр`)

Функция: `run_code_2(file_bytes)`.

Что делает:

- Находит пары листов `*Wd` и `*Md` (суффиксы последние 2 символа, регистр не важен).
- Группирует по префиксу.
- Для каждого префикса строит лист `<префикс>контр` (или `контр`) с итогами оплат и выполнений.
- После построения удаляет исходные `Wd/Md` листы из выходного файла.
- Дополнительно: строки, где одновременно `F=0` и `J=0`, не добавляются в выходной лист.

## Обработка “Запасы”

Функция: `run_code_3_inventory(file_bytes, accounts)`.

Что делает:

- Пользователь выбирает `1310/1320/1330`.
- По каждому счету ищется лист(ы), где в названии встречается номер счета.
- На листе ищется строка в колонке `A`, где ячейка равна номеру счета.
- Выше ставятся пороги в `J..N`, ниже заполняются формулы `J..N` до первой пустой в `G`.
- В строке с номером счета добавляются суммы по `J..N` (жирным, с заливкой).
- Числа форматируются как `#,##0`, рамки пунктиром, выравнивание по центру, шрифт 9.

## Streamlit Cloud: важные ограничения

- Streamlit Cloud не принимает `.xls` в uploader (ограничение платформы).
- Локально на Windows `.xls` поддерживается через конвертацию в `.xlsx` при наличии Excel + `pywin32`.

Рекомендация для Cloud:

- заранее конвертировать `.xls` в `.xlsx` и загружать уже `.xlsx`.

## Производительность

Места, которые специально оптимизированы:

- `copy_sheet(...)` копирует только реально существующие ячейки (`_cells`), а не весь прямоугольник `max_row * max_col`.
- Для реестров `Wr/Mr` высота строк выставляется только тем строкам, которые реально существуют (через `_cells` и `row_dimensions`), без цикла `1..max_row`.

Если обработка “тормозит”:

- чаще всего причина — очень большие листы ОСВ или много merged‑диапазонов,
- на локальной машине `.xls`‑конвертация через Excel COM тоже занимает заметное время.

## Частые ошибки и как чинить

`At least one sheet must be visible`

- Возникает при сохранении, если в книге нет видимых листов.
- В сборке добавлен служебный лист `Сборка`, который удаляется, когда добавились реальные листы.

`application/vnd.ms-excel files are not allowed`

- Это Streamlit Cloud: `.xls` заблокирован на загрузке. Используйте `.xlsx`.

`'MergedCell' object has no attribute ...`

- Обычно связано с копированием ячеек merged‑диапазонов.
- В `copy_sheet(...)` merged‑ячейки пропускаются.

## Как читать код

Опорные блоки в `app.py`:

- Вспомогательные: имена листов, уникализация, копирование листов.
- Сборка `_Анализ`: `build_analysis_workbook(...)`.
- Очистка ОСВ: блок `OSV Cleaning...`.
- Обработки:
  - `run_code_1` (сальдо + сальд (2))
  - `run_code_2` (контракты)
  - `run_code_3_inventory` (запасы)
- UI Streamlit: низ файла (загрузка, запросы префиксов, сборка, доступность сценариев, запуск обработок).

## Деплой

Для деплоя в Streamlit Cloud обычно достаточно:

- `app.py`
- `requirements.txt`

Но имейте в виду:

- `.xls` в Cloud не работает (см. ограничения выше).
- `pywin32` устанавливается только на Windows; в `requirements.txt` он должен быть зашит через marker по платформе.
